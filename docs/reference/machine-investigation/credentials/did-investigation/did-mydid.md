# did:mydid (myDid Method)

| Property | Value |
|----------|-------|

| DID Format | `did:mydid:<base58-encoded-ethereum-address>` |

## Overview

myDid is a decentralized identifier (DID) method that enables creation of DIDs and associated DID Documents on EVM (Ethereum Virtual Machine) compatible blockchains. The method uses ECDSA Secp256k1 cryptography for key generation and signature verification.

The identifier is generated by taking a public key, deriving the 20-byte Ethereum address from it, and encoding that address in base58. This results in a 58-character identifier.

DID Documents can exist in two forms:
- **On-chain**: Stored directly in the myDid smart contract, containing controllers, keys, and authentication methods
- **Off-chain**: Dynamically generated from the public key contained in the DID identifier when no on-chain data exists

Resolution involves extracting the Ethereum address from the identifier, querying the myDid smart contract, optionally retrieving off-chain content via URI references, and returning a merged view of both data sources.

## Evaluation

### 1. Feasibility/Complexity

**Cannot be supported without external dependencies.**

The did:mydid method fundamentally requires:
- Access to an EVM blockchain to query the smart contract for on-chain DID Documents
- Ability to make HTTP/RPC calls to blockchain nodes
- Optional IPFS or similar services for off-chain document references

There is no way to verify a did:mydid without querying the blockchain state. The DID identifier itself (the base58-encoded Ethereum address) does not contain enough information to reconstruct the DID Document or verify cryptographic material without blockchain access.

### 2. Ecosystem

**Very limited ecosystem.**

- GitHub repository has 0 stars, 0 forks, and 0 watchers
- Only 1 subscriber to the repository
- Organization has only 4 repositories total
- No known production deployments or integrations
- Not registered in the W3C DID Method Registry
- Reference implementation includes a Node.js resolver and React webapp, but adoption appears minimal

### 3. Stability

**Immature and potentially abandoned.**

- Repository created: August 3, 2023
- Last updated: August 17, 2023
- Last push: October 4, 2023
- No license specified
- No description provided
- Single README.md file serves as the specification
- No formal governance structure documented

## Special Considerations

- The method is essentially another Ethereum-based DID method similar to did:ethr, but with less adoption and tooling
- The specification acknowledges that "residual risks are the ones associated with any EVM-based solution: if the associated blockchain is compromised, the solution can become vulnerable"
- Security features include nonce-based replay attack prevention and signature-based authentication
- The 58-character identifier format is somewhat unusual compared to other Ethereum-based methods

## Recommendation

**No-go**

This DID method cannot be supported in our system due to the fundamental requirement for blockchain access to resolve and verify DIDs. Nodes would need to make HTTP/RPC calls to Ethereum (or other EVM) nodes to query the smart contract state, which violates our key constraint that nodes cannot make HTTP requests to validate DIDs.

Additionally, even if blockchain access were not a constraint, this method would not be recommended due to:
1. Extremely limited ecosystem adoption (zero community engagement)
2. Apparent abandonment (no updates since late 2023)
3. No formal specification beyond a basic README
4. No W3C registry listing
5. Better-established alternatives exist for Ethereum-based DIDs (e.g., did:ethr)
