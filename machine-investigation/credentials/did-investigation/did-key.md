# did:key (Key DID Method)

| Property | Value |
|----------|-------|
| Specification | [did:key Spec](https://w3c-ccg.github.io/did-key-spec/) |
| Organization | W3C Credentials Community Group |
| DID Format | `did:key:<multibase-multicodec-public-key>` |

## Overview

The did:key method is a non-registry DID method that encodes a cryptographic public key directly into the DID identifier. The DID Document is deterministically generated from the public key without any network lookups or registry access. This is one of the most widely adopted DID methods due to its simplicity.

Example DID: `did:key:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK`

### DID Structure

- **Prefix**: `did:key:`
- **Identifier**: Multibase-encoded (base58-btc `z` or base64-url `u`) + Multicodec (key type) + raw public key bytes

Format: `did:key:MULTIBASE(base58-btc, MULTICODEC(key-type, raw-public-key-bytes))`

### Supported Algorithms

| Algorithm | Multicodec | Key Length |
|-----------|-----------|------------|
| Ed25519 | 0xed | 32 bytes |
| secp256k1 | 0xe7 | 33 bytes (compressed) |
| X25519 | 0xec | 32 bytes |
| P-256 | 0x1200 | 33 bytes (compressed) |
| P-384 | 0x1201 | 49 bytes (compressed) |
| BLS12-381 | various | various |

### CRUD Operations

| Operation | Method |
|-----------|--------|
| Create | Generate key pair, encode public key with multibase+multicodec |
| Read | Deterministically expand DID to DID Document (no network needed) |
| Update | **Not supported** - key change = new DID |
| Deactivate | **Not supported** |

### Resolution Process

Resolution is purely deterministic - the DID Document is generated by:
1. Validating DID format
2. Decoding multibase-multicodec value
3. Verifying key length and validity
4. Generating verification methods
5. Constructing complete DID Document with relationships

## Evaluation

### 1. Feasibility/Complexity

**Highly feasible - Recommended**

The did:key method is ideal for our use case:

- **No external dependencies**: Completely self-contained
- **Deterministic resolution**: DID Document derived algorithmically from public key
- **No registry needed**: Node can verify without external queries
- **Simple implementation**: Well-documented, widely implemented
- **Multicodec flexibility**: Supports multiple key types via standard encoding

**Adaptation requirements**: None. We can implement did:key support directly by:
1. Supporting multibase/multicodec encoding
2. Implementing deterministic document generation for supported key types
3. Validating key material

### 2. Ecosystem

**Excellent / Industry standard**

- **W3C CCG**: Official W3C Credentials Community Group specification
- **Wide adoption**: Used across most DID/VC implementations
- **Universal Resolver**: Full support
- **Interoperability test suite**: Formal testing and compliance
- **Library support**: Available in most programming languages

### 3. Stability

**Stable and mature**

- **Version 0.9**: Mature specification
- **Active maintenance**: W3C CCG oversight
- **Backward compatible**: Multicodec ensures future key types work
- **Well-tested**: Extensive interoperability testing

## Special Considerations

### Limitations

- **No key rotation**: Key change = new DID entirely
- **No revocation**: Cannot mark a did:key as invalid
- **Short-lived use**: Recommended for ephemeral interactions only
- **Key compromise risk**: Compromised key cannot be rotated

### Best Use Cases

- Ephemeral peer-to-peer communications
- Development and testing
- Short-lived authentication sessions
- Bootstrap scenarios
- Cases where simplicity outweighs key management needs

### Design Benefits

- Zero infrastructure costs
- Complete offline operation
- Maximum simplicity
- Immediate availability
- Deterministic verification

## Recommendation

**Recommended**

The did:key method is highly suitable for our registry:

1. **No external dependencies**: Perfect for ledger verification - nodes can verify without HTTP calls
2. **Wide ecosystem support**: Maximizes compatibility with existing tools
3. **Simple to implement**: Well-documented algorithm
4. **Standard encoding**: Multibase/multicodec provides algorithm agility

**Implementation notes:**
- Support at minimum: Ed25519, secp256k1
- Consider also: P-256 (for enterprise compatibility)
- Can derive X25519 encryption keys from Ed25519 signing keys

**Limitations to document for users:**
- No key rotation - enterprise users needing rotation should use a different method
- Best suited for simpler use cases (single owner, no delegation)

This should be one of our primary supported DID methods.
